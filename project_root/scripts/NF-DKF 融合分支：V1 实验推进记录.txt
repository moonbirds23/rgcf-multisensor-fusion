# NF-DKF 融合分支：V1 实验推进记录（当前窗口总结）

## 一、当前阶段定位

本窗口的核心任务，是把项目从 **V0: post-only clean baseline** 往 **V1: post + meas 的 direct fusion** 推进一步，但严格遵守当前主线约束：

1. 不破坏 V0 clean baseline
2. V1 只验证 `meas-stream` 本身是否有价值
3. 当前不加 `gate`
4. 当前不加时间窗口 / temporal
5. 当前不改 loss
6. 当前不把更换 backbone 当创新重点

这些约束与当前实验推进计划一致：V0 是零号基线；V1 的定义是正式引入 `meas-stream`，不加 gate、不加时间窗口，采用最简单的 direct dual-stream fusion，用来回答“量测异构流是否值得正式引入”。
同时，当前项目主干也明确：主创新方向不优先押注 backbone，更关注 `meas-stream / soft gate / temporal-aware gate / reliability modelling`。

---

## 二、本窗口完成的总体工作

本窗口没有去大改整个训练框架，而是围绕 **V1 最小闭环** 做了以下推进：

1. 明确了 V1 的方法定位：
   `post-stream + meas-stream`，先做 **direct fusion**，不加 gate、不加 temporal。
2. 审查了当前 `post_features.py / meas_features.py / builders.py / dataset.py / trainer.py / model_config.py / model_factory.py / experiment_presets.py / runner.py` 的实际状态。
3. 确认当前训练逻辑已经从“现场生成训练数据”切换为“先生成 `dataset_store`，再从本地读取训练”。这一步是当前运行逻辑相较旧版最重要的变化之一。
4. 将 V1 的关键改动集中在：

   * `simulation/runner.py`
   * `features/meas_features.py`
   * `configs/model_config.py`
   * `models/gnn_fusion.py`
   * `models/model_factory.py`
   * `configs/experiment_presets.py`
5. 成功打通了 **V1 simulate 入口**，命令 `python main.py --mode simulate --preset clean_v1_post_meas_direct` 已经能正常运行并保存结果。
6. 同时识别出：**旧的 V0 dataset store 不能直接用于当前 V1 训练**，因为旧 raw sim schema 中缺少 `xpred` 等 V1 需要的字段，训练时已实测报错 `KeyError: 'xpred'`。

## 三、V1 的方法定义（本窗口最终统一口径）

当前对 V1 的统一理解如下：

### 1. V0

* 模型：`OriginalGNNFusion`
* 输入：仅 `post-stream`
* 特征核心：`xhat / diag(P) / valid`
* 本质：后验状态级融合器
* 不加 meas-stream
* 不加 gate
* 不加 temporal

### 2. V1

* 模型：`PostMeasDirectFusion`
* 输入：`post-stream + meas-stream`
* 做法：最简单的 direct dual-stream fusion
* 不加 gate
* 不加 temporal
* 不改 loss
* 目标：验证 `meas-stream` 是否有独立价值

这与项目文档一致：当前 original 的主要瓶颈在于它主要依赖 EKF 后验特征，因此更像“后验状态级融合器”，而不是“直接感知量测异构性和量测可靠性的融合器”；因此后续必须引入 `meas-stream`。

## 四、当前项目运行逻辑的关键更新（必须保留给后续窗口）

### 1. 训练流程已经不是旧语义

当前真实逻辑是：

1. 先通过 `generate_dataset` 生成 `dataset_store`
2. `dataset_store` 中保存 `train_sims / val_sims / test_sims / quick_sim`
3. 训练时从本地读取 `dataset_store`
4. 再动态从 `sim` 构造 `FeatureBundle`
5. 再由 `FusionTimeStepDataset` 构造 PyTorch Dataset

也就是说，当前 `dataset_store/` 保存的是 **raw sim store**，而不是已经切好的 feature tensor。
这是当前运行逻辑和旧描述最大的差异之一，文档中已经明确指出这一点。

### 2. features 层当前真实职责

当前目录说明中，`features/` 的职责是：

* `post_features.py`：后验状态流特征
* `meas_features.py`：量测异构流
* `builders.py`：统一组合 `post / meas`
* `dataset.py`：构造成 PyTorch Dataset

原说明里 `meas-stream` 还是接口预留，但本窗口推进的是把它真正变成 V1 的正式输入流。

## 五、本窗口涉及的具体代码改动（按文件说明）

下面是本窗口已经讨论并建议落地的代码层修改。这里按“目标、原状态、改后状态”来说明。

---

### A. `simulation/runner.py`

#### 原状态

旧版 `runner.py` 在一次 rollout 中主要记录：

* `xhat`
* `Phat`
* `valid_mask`
* `z_store`
* `R_store`

这与项目说明中的 runner 标准输出一致。

#### V1 改动目标

为了支撑标准的 `meas-stream`，仅有 `z_store / R_store` 不够，需要把 **update 前的预测统计量** 也记录下来，用于构造：

* innovation / residual
* whitened innovation
* NIS
* measurement geometry
* sensor type/meta

#### 本窗口建议/已实施的新增字段

在 `sim` 中新增以下字段：

* `xpred`：预测状态 `[K, N, 4]`
* `Ppred`：预测协方差 `[K, N, 4, 4]`
* `zpred_store`
* `innovation_store`
* `S_store`
* `Sdiag_store`
* `nis_store`
* `sensor_type_codes`
* `sensor_type_names`
* `sensor_pos`

#### 逻辑说明

在每个时间步、每个 sensor 上，流程改为：

1. `st_pred = ekf.predict(...)`
2. 生成 nominal measurement
3. fault 注入得到 `z_fin / R_fin / valid_fin`
4. 若有效，则：

   * 计算 `z_pred`
   * 计算 innovation
   * 计算 `S = H P^- H^T + R`
   * 计算 `NIS`
   * 再做 EKF update
5. 最终把这些统计量写进 `sim`

#### 意义

这一步使得 V1 不必走“弱化过渡版 meas-stream”，而是可以直接走 **标准 innovation/NIS 路线**。

---

### B. `features/meas_features.py`

#### 原状态

最开始 `meas_features.py` 只是占位逻辑：

* `node_feat = np.zeros((K, N, 0))`
* `enabled=False`
* 不真正参与建模

#### V1 改动目标

把 `meas-stream` 从空壳变成 **最小可用统计流**，并坚持项目文档中的原则：

* 不直接粗暴拼 raw measurement
* 优先在统计层/语义层对齐
* 第一版特征必须克制

这与实验推进计划中对 meas-stream 的推荐特征一致：优先纳入 `innovation / residual / normalized residual / NIS / diag(R) / valid / 少量 geometry / sensor type embedding`。

#### 本窗口建议/已实施的第一版 `Dm=13` 特征

当前建议的 13 维 `meas_feat` 包括：

1. `innov_slot0`
2. `innov_slot1`
3. `white_slot0`
4. `white_slot1`
5. `log1p_nis`
6. `log1p_Rdiag0`
7. `log1p_Rdiag1`
8. `dx_norm`
9. `dy_norm`
10. `range_norm`
11. `sensor_type_code_norm`
12. `valid_flag`
13. `meas_dim_like`

其中：

* 前 1~7 维是统计一致性信息
* 8~10 维是 geometry
* 11 维是 sensor type
* 12 维是 hard valid
* 13 维保留给量测维度语义

#### 重要说明

invalid 节点保留：

* type / geometry / valid

其余统计项默认置 0，避免 NaN。

### C. `features/builders.py`

#### 原状态

`build_feature_bundle_from_sim()` 已支持：

* 强制启用 `post-stream`
* 若 `bundle.model.use_meas_stream=True`，则构造 `meas`

#### 本窗口判断

这一层架构是对的，不是当前主要问题。
它已经能承担 V1 的桥梁作用：

* `post = build_post_node_features_from_sim(...)`
* `meas = build_meas_node_features_from_sim(...)`

#### 建议

后续可补：

* `post_dim / meas_dim` 元信息
* `shape assert`
* `post.mask == meas.mask` 的一致性检查

但这不是当前最急的阻塞点。

### D. `features/dataset.py`

#### 原状态

`FusionTimeStepDataset` 当前按**单时刻样本**组织数据：

* V0 返回：

  * `post_feat`
  * `mask`
  * `target`
* 若 `use_meas_stream=True`，则额外返回：

  * `meas_feat`

#### 本窗口结论

这一层已经具备 V1 兼容性，不需要重写。
这也和当前项目对学习式融合头的定位一致：虽然底层仿真和 EKF 是时序的，但当前学习式融合头仍然基本是**单时刻输入、单时刻输出**。

#### V0 / V1 dataset 差别的统一口径

* V0 dataset：`post_feat + mask + target`
* V1 dataset：`post_feat + meas_feat + mask + target`

差别不在样本粒度，而在样本内容。

### F. `models/gnn_fusion.py`

#### 原状态

已有 `OriginalGNNFusion`：

* 使用 post-only 输入
* 通过节点编码、attention、message passing、node logit 学权重
* 最后通过 `info-diag fusion` 融合 `xhat / pdiag`

#### 本窗口判断

`OriginalGNNFusion` 本身是合格的 V0 baseline，不应被魔改成双流混合体。
因为当前主线明确要求：V0 要保持为稳定、可复现、不可被后续模块破坏的零号基线。

#### V1 改动

新增类：

* `PostMeasDirectFusion`

其设计原则是：

1. 保留 V0 的后端融合骨架
2. 只在节点表示层引入 `meas-stream`
3. 不加 gate
4. 不加 temporal

#### 结构说明

* `post_enc(post_feat)`
* `meas_enc(meas_feat)`
* `concat -> fuse_proj`
* 后续继续用：

  * pairwise attention
  * message passing
  * node_logit
  * softmax weights
  * info-diag fusion

#### 核心思想

`meas-stream` 负责帮助模型学节点权重，而不是替代最终物理一致的 `info-diag` 融合形式。

* `original_gnn_fusion -> OriginalGNNFusion`

* 便于后续 V2/V3 扩展

### I. `experiments/generate_dataset.py`

#### 当前真实逻辑

`run_generate_dataset_experiment()` 的流程已经确认：

1. `build_sim_list_from_seed_range(...)` 生成：

   * `train_sims`

#### 重要结论

### J. `data/dataset_store.py`

#### 当前真实逻辑

`dataset_store.py` 当前负责：

* 生成 dataset_id
* 创建 dataset_store 目录
* 保存：

  * `train_sims.pkl`
  * `val_sims.pkl`
  * `test_sims.pkl`
  * `quick_sim.pkl`
* 写入 `meta.json`
* 根据 `dataset_id` 或自动匹配规则查找 store
* 从本地加载 raw sims

这与项目主干说明一致。

#### 当前最重要的认识

`dataset_store` 保存的是 **raw sim schema**。
因此，只要 `runner.py` 的 `sim` 字段结构变了，哪怕 `dataset_store.py` 本身没改很多，**dataset 的“内容语义”也已经变化了**。

#### 当前问题

`DatasetStoreMeta.feature_version` 仍是：

* `raw_obs_v1`

但现在 V1 已经引入了新 schema，旧版 `raw_obs_v1` 语义明显不够准确。

#### 本窗口建议

后续把它升级为类似：

* `raw_sim_v2_with_meas_stats`

并且把 `feature_version / schema_version` 纳入 `_meta_matches_bundle()` 的匹配逻辑。
否则以后很容易出现：

* preset 一样
* seed range 一样
* fault mode 一样
* 但内部 sim schema 不一样
  从而误加载旧 dataset。

## 七、当前对 dataset / 数据库（dataset_store）的统一理解

这里必须写清楚，防止后续窗口混淆。

### 1. 当前项目中的“dataset”分两层

#### 第一层：离线层（dataset_store）

保存在 `dataset_store/` 中，内容是：

* `train_sims.pkl`
* `val_sims.pkl`
* `test_sims.pkl`

这是 **raw sim 仓库**。

#### 第二层：在线层（feature dataset）

训练时才动态构造：

* `build_feature_bundle_from_sim()`
* `FusionTimeStepDataset`

这是 **真正喂给 PyTorch 的训练样本**。

### 2. V0 与 V1 的 dataset 差异

#### V0

单样本：

* `post_feat`
* `mask`
* `target`

#### V1

单样本：

* `post_feat`
* `meas_feat`
* `mask`
* `target`

### 3. 为什么旧 dataset 不能直接拿来训练 V1

因为旧 `dataset_store` 中保存的 raw sim schema 不包含：

* `xpred`
* `Ppred`
* `innovation_store`
* `S_store`
* `nis_store`
  等 V1 需要的字段。

## 十、后续窗口接力建议（优先级顺序）
